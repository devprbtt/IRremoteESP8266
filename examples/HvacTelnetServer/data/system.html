Third-Party Integration API

Overview

IR Server Telnet exposes two integration paths: a telnet JSON command socket for runtime control and a small HTTP API for configuration, status, diagnostics, and IR learning.

TELNET :4998 HTTP /api/* HTTP /devices

Telnet JSON Commands

  • list: returns emitters and configured device summaries
  • get: returns one device state by id
  • get_all: returns all current device states
  • send: sends an HVAC or custom command to one device
  • push_all: pushes all current states to connected telnet clients
  • raw: sends ad-hoc IR payloads by encoding
{"cmd":"list"} {"cmd":"get","id":"1"} {"cmd":"get_all"} {"cmd":"send","id":"1","power":"on","mode":"cool","temp":24,"fan":"auto","light":"true"} {"cmd":"send","id":"5","command_name":"power_on"} {"cmd":"raw","emitter":0,"encoding":"pronto","code":"0000 006D 0000 0022 ..."}

HTTP API Endpoints

  • GET /api/device/get?id=1: current state for one device profile
  • GET /api/device/get_all: current state for all device profiles
  • POST /api/device/send: send an HVAC command or custom command
  • POST /api/device/raw: send raw IR without a saved device profile
  • GET /api/status: live runtime status, memory, network, counts, DINplug usage
  • GET /api/meta: UI metadata, capabilities, and limits
  • GET /api/config: current saved configuration JSON
  • POST /api/config/save: replace saved configuration JSON
  • GET /api/diagnostics: latest persisted diagnostics snapshot with trend history
  • GET /api/wifi/scan: available WiFi networks
curl http://ir-server.local/api/device/get?id=1 curl http://ir-server.local/api/device/get_all curl http://ir-server.local/api/meta curl http://ir-server.local/api/config curl http://ir-server.local/api/diagnostics curl http://ir-server.local/api/wifi/scan curl -X POST http://ir-server.local/api/config/save \ -H "Content-Type: application/json" \ --data @config.json

IR Learn API

These endpoints are used by the device UI and can also be called directly if the IR receiver is enabled in Config.

  • POST /api/ir/learn/start with encoding set to pronto, gc, racepoint, or rawhex
  • GET /api/ir/learn/poll to check progress and collect the captured code
  • POST /api/ir/learn/cancel to abort a pending learn session
curl -X POST http://ir-server.local/api/ir/learn/start \ -d "encoding=pronto" curl http://ir-server.local/api/ir/learn/poll curl -X POST http://ir-server.local/api/ir/learn/cancel

HTTP Control Examples

These routes mirror the telnet control API and support both standard HVAC profiles and custom command profiles.

curl -X POST http://ir-server.local/api/device/send \ -H "Content-Type: application/json" \ -d '{"id":"1","power":"on","mode":"cool","temp":24,"fan":"auto","light":"true"}' curl -X POST http://ir-server.local/api/device/send \ -H "Content-Type: application/json" \ -d '{"id":"5","command_name":"power_on"}' curl -X POST http://ir-server.local/api/device/raw \ -H "Content-Type: application/json" \ -d '{"emitter":0,"encoding":"pronto","code":"0000 006D 0000 0022 ..."}'

Response Examples

Standard HVAC devices return runtime state. Custom profiles return the registered command catalog plus the command sent most recently.

{"type":"state","id":"1","protocol":"MITSUBISHI_AC","custom":false,"power":"on","mode":"cool","setpoint":24,"current_temp":24,"fan":"auto","light":"on"} {"type":"state","id":"5","protocol":"CUSTOM","custom":true,"profile_name":"TV","custom_commands":[{"name":"power_on","encoding":"pronto"},{"name":"volume_up","encoding":"pronto"}],"encoding":"pronto","command_name":"power_on"}

Integration Notes

  • Each telnet command must be one JSON object per line, terminated with CRLF or LF.
  • On client connect or reconnect, current device states are pushed automatically.
  • State updates caused by DINplug or the web UI are also broadcast to telnet clients.
  • If a web password is enabled, third-party HTTP callers must authenticate the same way as the UI.
  • For a fuller written reference, the same content is documented in the project README.
